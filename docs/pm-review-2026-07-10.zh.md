## 二、四维度评估

### 维度 1：用户体验

#### 提升表现（明确正向）

| 体验设计 | 具体表现 | 价值 |
|----------|----------|------|
| **默认 manual 安全策略** | 投递三档中默认不自动发送，auto 需三重门禁（配置 + 风险确认 + `--execute`） | 降低用户对"账号被封"的焦虑，建立信任 |
| **渐进式复杂度暴露** | manual → semi → auto，能力按需解锁 | 符合"先可用再好用"的认知曲线 |
| **一条命令离线 Demo** | `bash scripts/demo.sh` 无需填简历/登录即可看到完整产出 | 冷启动成本极低，转化漏斗顶部友好 |
| **人话一页摘要** | `brief_zh` 用口语化建议替代冷指标（"还有 N 个词没出现；会的就补，不会的别编"） | 降低指标理解门槛，强化"诚实"品牌调性 |
| **setup 粘贴旧简历冷启动** | `/setup-zh` 支持粘贴旧简历提取画像 | 减少首次填表摩擦 |
| **每日工作台** | `tracker.py today` 按面试/跟进/已结束分组 | 信息聚合，减少用户在多记录间跳转 |

#### 体验短板（需改进）

| 短板 | 现状 | 影响 |
|------|------|------|
| **强依赖命令行 + AI Agent** | 核心流程需在 Agent 中触发命令或在终端执行 Python 脚本 | 目标用户（普通求职者）门槛过高，实际可触达人群收缩为"会用 Agent 的开发者" |
| **工具分散，缺少统一编排入口** | 用户需在 5 个工具间切换：`install_domestic_search` → `apply_assist` → `match_resume` → `tracker` → `export_resume_pdf` | 认知负荷高，断点易流失；每步切换都是一次"是否继续"的决策 |
| **semi 模式跨平台剪贴板不一致** | macOS 用 `pbcopy`、Linux 用 `xclip/wl-copy`，Windows 无原生支持（降级为打印文本让用户手动复制） | Windows 用户（国内主流）体验断裂，半自动退化为"手动复制" |
| **匹配分缺乏语义理解** | TF-IDF 关键词对齐，"高并发"与"大流量"、"微服务"与"分布式架构"无法关联 | 用户看到 miss 列表里有自己实际掌握的技能，产生"工具不准"的不信任 |
| **无 GUI / Web 界面** | 纯 CLI + Agent 指令 | 与 Reactive-Resume / open-resume 所见即所得的体验差距明显 |

#### 用户体验结论

> 当前体验在"信任建立"和"冷启动"上表现优秀，但在"持续使用流畅度"上存在结构性短板：**工具链碎片化**是最大体验瓶颈，命令行门槛将目标人群锁死在窄带技术用户。

### 维度 2：功能完整性与业务逻辑

#### 完整度评估（链路覆盖）

```
搜岗 → 生成 → 匹配 → 投递 → 追踪 → 复盘
 ✓     ✓     ✓     ✓     ✓     △(diff 飞轮已有，复盘引导弱)
```

全链路基本闭环，但存在以下**功能缺陷与逻辑缺口**：

| 严重度 | 缺陷 | 根因分析 | 影响 |
|--------|------|----------|------|
| **P0** | `apply_assist.py` 的 `auto-greet` 命令**话术文件未真正传入 boss 命令** | 第 374 行 `cmd = ["boss","greet",sid]`，`text_file` 仅生成 `cmd_note` 注释字符串，未拼入实际执行命令 | auto 模式下自定义话术功能名存实亡；用户配置了话术但实际发送的是 boss-cli 默认内容 |
| **P0** | 根目录 `salary_lookup.py` 是**上游丹麦薪资逻辑**，国内不可用 | 处理 `A/S`、`ApS` 后缀与 `øæå` 字符，`salary_data.json` 需用户提供但无国内数据源 | 产品宣传含薪资查询，实际国内用户无法使用；catalog 的 `salary-negotiate` 仅为方法论 |
| **P1** | **匹配引擎 IDF 退化** | `build_idf` 仅基于简历 + JD 两篇文档（n=2）计算逆文档频率，IDF 区分度极弱，实际退化为 TF 加权 | "逆文档频率"的统计假设不成立，匹配分更多反映词频而非关键词重要性；与 Resume-Matcher 的 embedding 语义匹配差距大 |
| **P1** | **岗位信息未结构化** | JD 的薪资/地点/学历/经验要求未抽取为字段，tracker schema 无对应列 | 无法按薪资范围、城市筛选投递记录；无法做"投递漏斗按城市分布"等统计 |
| **P1** | **搜岗结果无法流入 tracker** | `install_domestic_search` 与 `tracker` 之间无数据管道，搜到的岗位需手动 `add` | 搜岗与投递脱节，用户搜到 N 个岗位后逐条手动录入，摩擦大 |
| **P2** | **无面试日程管理** | tracker 有 `interview` 状态但无日历/提醒集成 | 用户面试时间靠记忆或外部日历，工具内无主动提醒 |
| **P2** | **复盘引导弱** | `match_resume diff` 质量飞轮已有，但无 `/outcome` 与 diff 的自动串联 | 用户改完简历不知道该跑 diff 对比；质量飞轮需要用户主动发现 |

#### 业务逻辑合理性

**合理的设计决策**：
- CSV 作为唯一真相源：简单、可 diff、可版本管理，求职场景数据量级（百条）下足够
- 不自动投递政策：合规（PIPL）+ 风控（平台 ToS）双考量，与 AIHawk 的激进策略形成差异化
- catalog 成本卡透明化：`setup_cost` 字段明确标注搭建成本，避免"假装开箱可用"

**存疑的逻辑**：
- `_parse_yaml_lite` 手写 YAML 解析器（apply_assist.py 第 53-85 行）：虽有 `try import yaml` 降级，但手写解析器边界 case 多，是技术债而非必要自研
- auto 模式 `after-generate` 不会静默发送（第 436 行）：设计意图安全，但与用户"配置了 auto 就期望自动"的心智模型冲突，需更明确的引导

#### 功能完整性结论

> 链路闭环但存在 **2 个 P0 功能缺陷**（auto 话术断裂 + 薪资工具国内不可用）和 **3 个 P1 逻辑缺口**（IDF 退化 + JD 未结构化 + 搜岗追踪脱节）。匹配引擎的统计假设在小样本下不成立，是技术深度的核心短板。

### 维度 3：性能与稳定性

| 风险点 | 位置 | 严重度 | 说明 |
|--------|------|--------|------|
| **CSV 全量重写** | `tracker.py` `write_rows` | 低 | 每次 add/update 重写全部行；求职场景百条级数据可接受，但千条以上有性能与并发风险（无文件锁） |
| **PHRASE_TERMS 全文 replace** | `match_resume.py` `tokenize` | 低 | 多次 `str.replace` 扫描全文，长 JD（万字）性能可接受但不优雅；正确做法是先 tokenize 再合并 |
| **subprocess 调用外部 boss-cli** | `apply_assist.py` `cmd_auto_greet` | 中 | `boss` 命令接口变更即失败；且当前 cmd 写死 `["boss","greet",sid]`，话术未传入，功能本身已断裂 |
| **HTML 拼接 XSS** | `tracker.py` `export_html` | 低 | 已用 `html.escape`，风险可控；但 f-string 拼接结构脆弱，后续扩展易引入遗漏 |
| **手写 YAML 解析** | `apply_assist.py` `_parse_yaml_lite` | 中 | PyYAML 缺失时降级，但解析器逻辑对缩进/引号/注释的处理可能存在边界 case 未覆盖 |
| **无并发控制** | 全局 | 低 | 单用户本地工具，无并发场景；但若未来做 Web 化需引入文件锁/DB |
| **auto 模式无速率限制** | `apply_assist.py` | 中 | `max_batch: 5` 配置存在但 `cmd_auto_greet` 未实际使用该字段做批量节流 | 

#### 性能稳定性结论

> 当前数据量级下性能无瓶颈。主要风险集中在 **auto 模式的功能断裂**（话术未传入 + max_batch 未生效）和 **外部依赖脆弱性**（boss-cli 接口变更无兜底）。手写 YAML 解析器是可消除的技术债。

### 维度 4：竞品差异化

#### 差异化优势（护城河）

| 优势 | 本项目 | 竞品普遍做法 | 价值评估 |
|------|--------|--------------|----------|
| **本地优先 + PIPL 合规** | 简历/投递数据本地存储，敏感文件 gitignore | 在线 AI 简历站数据上云 | 隐私敏感用户的核心选择理由 |
| **默认不自动投递** | manual 为默认，auto 三重门禁 | AIHawk/EasyApplyBot 主打全自动海投 | 合规差异化，规避封号风险 |
| **Agent skill 可 fork** | 技能 + 命令框架，可改可扩展 | 封闭产品 | 开发者用户群的可定制性 |
| **catalog 成本透明** | `setup_cost` 字段明确标注 | 竞品宣传重功能但隐藏搭建成本 | 诚实建立信任，降低用户期望落差 |
| **诚实能力清单** | README 区分"已落地"和"明确不做" | 多数项目只列功能不列边界 | 品牌调性差异化 |

#### 差异化不足（竞争劣势）

| 不足 | 本项目 | 竞品标杆 | 差距 |
|------|--------|----------|------|
| **无 GUI/Web** | 纯 CLI + Agent | Reactive-Resume / open-resume 所见即所得 | 体验代差，非技术用户无法使用 |
| **无语义匹配** | TF-IDF 关键词对齐 | Resume-Matcher embedding 余弦相似度 | 匹配精度差距，miss 列表可信度低 |
| **无端到端投递** | 默认手动投递 | claude-apply CDP 真实浏览器填写 | 投递效率差距（但属有意取舍） |
| **社区规模小** | 新项目 | career-ops 59k★ / Resume-Matcher 27k★ | 缺乏社会证明，分发冷启动难 |
| **无多用户/协作** | 单机本地 | — | 无法做团队/班级场景（但可能非目标） |
| **薪资数据空白** | 丹麦遗留代码 + catalog 仅方法论 | salary-compare 含五险一金/加班/通勤 | 国内求职核心决策维度缺失 |

#### 竞品结论

> 差异化优势集中在"合规 + 诚实 + 可定制"，是隐私敏感型和技术型用户的有效切入点。但"无 GUI + 无语义匹配 + 无薪资数据"三重短板，限制了用户规模与决策价值。端到端投递的取舍是正确的（合规优先），不应盲目追赶。

## 三、迭代建议（按优先级排列）

### 优先级矩阵说明

- **P0**：功能缺陷修复，影响产品可信度，立即处理
- **P1**：体验与逻辑缺口，影响留存与转化，1–2 周内
- **P2**：中长期能力建设，影响规模化，1–3 月

---

### 短期快速优化（1–2 周）

#### P0-1 修复 auto-greet 话术文件功能断裂
- **问题**：`apply_assist.py:374` 的 `cmd` 未包含 `text_file`，话术配置名存实亡
- **方案**：探测 boss-cli 是否支持自定义话术参数（如 `--message`/`--text`），支持则拼入；不支持则在 dry-run 输出中明确提示"当前 boss-cli 版本不支持自定义话术，请改用 semi 模式粘贴"
- **预期价值**：消除功能虚假承诺，避免用户配置后产生错误预期
- **是否自研**：否，修复现有逻辑，不引入新依赖

#### P0-2 处理 salary_lookup.py 国内不可用问题
- **问题**：根目录薪资工具是丹麦逻辑，国内用户无法使用
- **方案**：二选一——
  - (a) 将 `salary_lookup.py` 移入 `integrations/legacy/` 并标注"上游丹麦功能，国内待实现"
  - (b) 接入国内薪资数据源（如职友集/看准网公开数据），改造为国内版
- **推荐**：(a) 先止血标记，(b) 纳入 P2 规划
- **预期价值**：消除产品宣传与实际的落差

#### P1-1 统一编排入口，减少命令切换
- **问题**：用户需在 5 个工具间手动切换
- **方案**：新增 `tools/flow.py`，提供 `apply <JD>` 一条命令串联：生成 → 匹配 report → PDF 导出 → suggest-add 提示
- **可复用轮子**：无需新依赖，编排现有 4 个工具
- **是否自研**：是（编排层，非重复造轮子——现有工具各自独立，无统一调度）
- **预期价值**：核心闭环从 5 步降为 1 步，显著降低断点流失

#### P1-2 JD 结构化解析
- **问题**：岗位薪资/地点/学历/经验未抽取为结构化字段
- **方案**：在 `match_resume.py` 新增 `parse-jd` 子命令，正则 + 规则抽取薪资区间、城市、学历要求、经验年限，输出 JSON；tracker schema 增加 `salary`、`city`、`education`、`experience` 列
- **可复用轮子**：可参考 `spencergg/resume-parser`（ResumeSDK）的解析思路，但 JD 解析无成熟开源库，需自研轻量规则
- **是否自研**：是（JD 解析为国内特有场景，无直接可用轮子）
- **预期价值**：解锁按城市/薪资筛选与统计，提升 tracker 决策价值

#### P1-3 匹配引擎 IDF 优化
- **问题**：n=2 的 IDF 退化为 TF 加权
- **方案**：内置一份预计算的技能语料 IDF（基于公开 JD 语料库统计），替换动态 build_idf；或引入轻量同义词扩展（高并发↔大流量、微服务↔分布式）
- **可复用轮子**：可参考 Resume-Matcher 的关键词抽取逻辑（Apache-2.0），但 embedding 层保持 catalog 可选
- **是否自研**：是（轻量语料 IDF，不引入 embedding 依赖）
- **预期价值**：提升匹配分可信度，减少"明明会但显示 miss"的不信任

#### P1-4 semi 模式 Windows 剪贴板补充
- **问题**：Windows 无 `pbcopy`，降级为打印文本
- **方案**：`copy_to_clipboard` 增加 Windows 分支：`subprocess.run(["powershell","-Command","Set-Clipboard","-Value",text])` 或 `clip.exe`
- **是否自研**：否，补充平台适配
- **预期价值**：Windows 用户（国内主流）半自动体验从"手动复制"恢复为"一键复制"

### 中长期规划（1–3 月）

#### P2-1 轻量 Web 看板（只读可视化 + 状态更新）
- **问题**：纯 CLI 限制非技术用户
- **方案**：基于 tracker CSV，用 FastAPI + 内嵌静态页提供只读看板（复用现有 `export_html` 逻辑）+ 状态更新 API；不引入前端框架，单文件 HTML + fetch
- **可复用轮子**：FastAPI（成熟）、现有 `export_html` 的 HTML 模板
- **是否自研**：是（轻量编排，非重复造轮子——Reactive-Resume 太重，不适合只读看板场景）
- **预期价值**：降低非技术用户门槛，为 P2 分发扩大用户基数

#### P2-2 搜岗结果流入 tracker（岗位聚合管道）
- **问题**：搜岗与投递脱节
- **方案**：`install_domestic_search` 搜岗结果输出标准化 JSON，新增 `tracker.py import-jobs <json>` 批量导入并去重
- **预期价值**：搜岗→投递链路打通，减少手动录入

#### P2-3 国内薪资数据接入
- **问题**：薪资维度空白
- **方案**：接入职友集/看准网公开薪资数据（需评估数据源合规性），或提供"用户自填薪资期望 + JD 抽取薪资范围"的轻量对比
- **可复用轮子**：参考 `JayLyu/salary-compare` 的对比维度（五险一金/加班/通勤/时薪）
- **预期价值**：补全求职核心决策维度

#### P2-4 语义匹配可选层（catalog 化）
- **问题**：TF-IDF 缺乏语义理解
- **方案**：将 sentence-transformers 本地 embedding 作为 catalog 可选层，用户按需启用；默认仍用 TF-IDF
- **可复用轮子**：Resume-Matcher（Apache-2.0）的匹配逻辑
- **是否自研**：否，复用现有轮子，catalog 化
- **预期价值**：匹配精度可选升级，不破坏默认零依赖路径

#### P2-5 P2 分发：社区内容与一键安装
- **问题**：社区规模小，分发冷启动难
- **方案**：Agent 一键安装脚本（覆盖 Claude Code / Cursor / 国产模型 Agent）；小红书/掘金技术内容；Issue 赛道模板已有，补充"用户故事"类内容
- **预期价值**：建立社会证明，扩大用户基数

## 五、总结

**当前方案的核心价值**在于"合规优先 + 诚实能力清单 + 可 fork 的 Agent 框架"，这一组合在同赛道中具有明确的差异化定位。Phase 1–3 的工程完成度扎实，catalog 治理机制（allowlist + 成本卡）是值得肯定的产品决策。

**最紧迫的风险**是 2 个 P0 功能缺陷（auto 话术断裂 + 薪资工具国内不可用），它们直接损害产品可信度。**最关键的增长瓶颈**是工具链碎片化——5 个独立命令构成的闭环，在"持续使用流畅度"上存在结构性短板，统一编排入口（P1-1）是性价比最高的改进。

**中长期方向**应聚焦于"降低使用门槛"（Web 看板）和"补全决策维度"（薪资数据 + 语义匹配），而非盲目追赶竞品的端到端自动投递——合规取舍是本项目的护城河，不应动摇。
